## fls意识到问题
问题1：覆盖 triggerAntiCheat 函数
因为 triggerAntiCheat 是全局函数（挂在 window 上），Tampermonkey 脚本可以直接用 triggerAntiCheat = function() {...} 覆盖它，把切屏计数改成永远是0或-1，永远不触发提交。
问题2：按P看答案
因为 S.activeExam.questionsList 包含 correct_answer 字段，存在前端内存里，任何脚本都能读到。服务器把正确答案直接发给了客户端。
问题3：覆盖 teacherLogin 伪装成admin
同理，teacherLogin 是全局函数，可以被覆盖。而且教师验证完全在前端——只要把 S.teacher.id 设成 teacher-1 就能冒充admin。

根本原因：所有逻辑都在前端，前端不可信。
修法：
问题2（最关键）：答案不能发给前端。
服务器的 /api/questions/:exam_id 现在返回 correct_answer，要改成学生身份请求时不返回这个字段，提交时服务器自己算分，不信任前端传来的分数。
问题1：全局函数被覆盖。
把关键变量和函数封装进立即执行函数（IIFE），不挂到 window 上，脚本就覆盖不了。但这治标不治本，因为 Tampermonkey 有 unsafeWindow 访问权限。
问题3：教师登录在前端伪造。
服务器要返回一个 session token，之后每次请求都验证 token，而不是信任前端传来的 teacher_id。

实际上能完全修好的只有问题2，把答案从前端移走，按P就看不到答案了。问题1和3从浏览器层面很难完全防住 Tampermonkey，因为它本来就是设计来改网页的。
你想先修哪个？答案泄露是最严重的，建议先修这个。

## fls意识到token
登录时服务器生成随机 token 存在内存里，返回给前端。之后所有教师 API 请求都带上这个 token，服务器验证。
就算学生覆盖 teacherLogin 绕过前端登录，S.token 是 null，所有教师 API 都会返回 401，拿不到任何数据。只有真正通过服务器验证密码拿到合法 token 才能操作。
注意： token 存在服务器内存里，服务器重启后所有 token 失效，老师需要重新登录。
